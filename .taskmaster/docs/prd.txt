# PDFParse — Product Requirements Document
# Lightweight PDF Text Extraction Desktop App

## 1. Overview

PDFParse is a lightweight desktop application for extracting text from PDF files.
Built with Svelte (frontend) and Tauri (backend/desktop shell), it uses
@llamaindex/liteparse for spatial text extraction from PDFs. LiteParse preserves
the original layout (column alignment, indentation) through spatial text parsing.
OCR and Office document support are intentionally disabled — PDF-only, text-layer
extraction.

## 2. Goals

- Provide a fast, offline, cross-platform desktop tool to extract text from PDFs
- Keep the app minimal — no OCR, no Office document support, no cloud services
- Deliver a clean, intuitive single-window UI

## 3. Non-Goals

- OCR (scanned image PDFs are out of scope)
- Office document parsing (docx, xlsx, pptx, etc.)
- Cloud upload / server-side processing
- PDF editing or annotation
- Batch processing (v1 — single file at a time)

## 4. Target Users

- Knowledge workers who need to quickly grab text from PDF reports
- Developers who want a lightweight local tool instead of online converters
- Anyone who values privacy and offline-first tooling

## 5. Core Features

### 5.1 File Selection
- Open a file picker dialog to select a single PDF file
- Drag-and-drop a PDF onto the app window
- Display the selected file name and size

### 5.2 Text Extraction
- Parse the PDF using @llamaindex/liteparse (OCR disabled)
- Extract all text content page by page with spatial layout preserved
- Show a progress indicator during extraction
- Handle errors gracefully (encrypted PDFs, image-only PDFs, corrupt files)

### 5.3 Text Display
- Show extracted text in a scrollable, read-only text area
- Display page separators (e.g., "--- Page 1 ---")
- Show total page count and character count

### 5.4 Text Export
- Copy all text to clipboard with one click
- Save extracted text to a .txt file via save dialog
- Preserve page structure in exported text

## 6. UI Design — "Ink & Paper" Neo-Editorial

### 6.1 Design Concept

Aesthetic direction: luxury publishing meets precision tool. The app should feel
like a premium document workshop — not a generic dashboard. Inspired by
letterpress printing, fine stationery, and editorial typography.

Key differentiator: extracted text pages look and feel like actual printed sheets
with ruled-line texture, and a "printing press" extraction animation.

### 6.2 Layout

Split-panel layout with compact sidebar navigation:

┌────┬──────────────────────────────────┐
│    │  Compact Title Bar               │
│ S  ├──────────────────────────────────┤
│ I  │                                  │
│ D  │  Main area:                      │
│ E  │  - Drop zone (empty state)       │
│ B  │  - Text viewer (loaded state)    │
│ A  │                                  │
│ R  ├──────────────────────────────────┤
│    │  Status bar + Actions            │
└────┴──────────────────────────────────┘

Sidebar (48-56px wide):
- App logo/icon at top
- File stats displayed vertically (page count, char count, file size)
- Compact icon-only navigation
- Collapses to icon-strip on narrow windows

### 6.3 Typography

Load via Google Fonts or bundle locally:

| Role           | Font                    | Fallback            |
|----------------|-------------------------|---------------------|
| Display/title  | Cormorant Garamond 600  | Georgia, serif      |
| Body/UI        | DM Sans 400/500/700    | system-ui, sans     |
| Extracted text | JetBrains Mono 400     | monospace           |
| Stats/labels   | DM Sans 500 (small caps)| system-ui, sans    |

- Display font is editorial serif — unusual for a tool app, makes it memorable
- Body sans is geometric and clean for UI elements
- Monospace for extracted content preserves spatial layout integrity

### 6.5 Visual Effects & Texture

- Paper grain: subtle CSS noise texture overlay on the main content area
  (generated via SVG filter or tiny repeating PNG, opacity 0.02-0.04)
- Ruled lines: faint horizontal lines in the text display area (like notebook
  paper), created with repeating-linear-gradient, line-height matched
- Page cards: each extracted page rendered as a card with:
  - Subtle paper texture background
  - Very slight box-shadow (like a sheet of paper sitting on a desk)
  - Top-left corner fold effect (CSS triangle, subtle)
  - Page number as a "folio" in the top-right corner (editorial style)
- Drop zone: large centered area with a dashed border using a custom dash
  pattern (SVG stroke-dasharray), animated on hover/drag
- Drop zone icon: stylized PDF document icon with folded corner, drawn in CSS
  or inline SVG

### 6.6 Animation & Motion

- **Page load stagger**: When extraction completes, pages appear one by one
  with a staggered slide-up + fade-in (animation-delay: index * 60ms).
  Simulates pages coming off a printing press.
- **Extraction progress**: Instead of a generic spinner, show a horizontal
  "ink line" that grows across the status bar (like ink spreading on paper).
  Use a CSS animation with a shimmer gradient.
- **Drop zone hover**: The dashed border segments rotate slowly (CSS animation
  on stroke-dashoffset). On drag-over, the icon scales up and the border
  color transitions to accent.
- **Button micro-interactions**: Buttons have a subtle press-down effect
  (translateY(1px) + darker bg) on :active, not just hover.
- **Toast notifications**: Slide in from bottom-right with a slight bounce
  easing. Auto-dismiss with a shrinking progress bar at the bottom.
- **Page card hover**: Very subtle lift (translateY(-2px)) with shadow
  increase on individual page cards.

### 6.7 Component Details

**Sidebar**
- Fixed width 52px, dark-toned regardless of theme (provides contrast)
- App icon: custom SVG — stylized "P" in a document shape
- Stats: vertical list with small-caps labels + bold values
- Active state: thin 2px accent line on the left edge

**Header / Title Bar**
- Compact: single row with file name (truncated with ellipsis) + extraction
  status badge (idle / extracting / done / error)
- Status badge: pill shape, color-coded, with subtle pulse animation when
  extracting

**Drop Zone (empty state)**
- Full-height centered panel
- Large serif heading: "Drop your PDF"
- Subtitle: "or click to browse — text extraction in seconds"
- PDF icon: 64px, monochrome, with folded corner detail
- Dashed border: 2px, rounded 16px, accent color on drag

**Text Viewer (loaded state)**
- Scrollable area with page cards
- Each card: cream background, ruled lines, page folio number
- Custom scrollbar: thin (6px), accent-colored thumb, rounded
- "Jump to page" mini-navigation in the sidebar when multiple pages

**Status Bar / Footer**
- Left: status text (muted) + extraction progress indicator
- Right: action buttons — "Copy All" (outline) + "Save TXT" (filled accent)
- Compact height: 48px

**Error States**
- Error banner: full-width below header, vermillion left border (4px),
  cream background, icon + message + dismiss button
- Not a modal — doesn't block interaction

**Toast**
- Position: bottom-right, 16px from edges
- Shape: rounded 12px, with left color-band (4px, success/error color)
- Auto-dismiss: 3 seconds with visible countdown bar

### 6.8 Responsive Behavior

- >= 800px: full sidebar + split layout
- 600-799px: sidebar collapses to thin icon strip (36px)
- < 600px: sidebar hidden, stats move to collapsible header section

### 6.9 Accessibility

- All interactive elements keyboard-navigable
- Focus rings: 2px offset accent outline (visible on both themes)
- WCAG AA contrast ratios for all text/background combinations
- Screen reader labels for icon-only sidebar buttons
- Respect prefers-reduced-motion: disable stagger animations, use opacity-only

## 7. Technical Architecture

### Frontend (Svelte)
- Framework: Svelte 5 (latest)
- Bundler: Vite
- Styling: Plain CSS or Tailwind CSS (keep it simple)
- State: Svelte stores for app state

### Backend (Tauri + Rust + Node.js sidecar)
- Tauri v2 for desktop shell
- @llamaindex/liteparse for PDF text extraction (OCR disabled)
- Node.js parser script compiled to standalone binary via `pkg` and bundled
  as a Tauri sidecar (no Node.js installation required on user's machine)
- Tauri commands exposed to frontend:
  - `extract_text(file_path: String) -> Result<ExtractionResult, String>`
    (internally calls sidecar binary)
  - `save_text(content: String, path: String) -> Result<(), String>`
- Tauri shell plugin for sidecar execution
- Tauri dialog plugin for file open/save dialogs
- Tauri clipboard plugin for copy functionality

### Data Flow
1. User selects PDF → frontend sends file path to Rust backend
2. Rust invokes liteparse sidecar → sidecar outputs JSON (per-page text)
3. Rust parses sidecar output → returns structured result to frontend
4. Frontend renders text → user can copy/save

## 8. Error Handling

| Scenario              | Behavior                                      |
|-----------------------|-----------------------------------------------|
| Encrypted PDF         | Show "This PDF is password-protected" message |
| Image-only PDF        | Show "No extractable text found" message      |
| Corrupt file          | Show "Unable to read this file" message       |
| Non-PDF file          | File picker filters to *.pdf only             |
| Very large PDF (>100MB)| Show warning, proceed if user confirms       |

## 9. Performance Requirements

- Extract text from a 100-page PDF in under 3 seconds
- App startup in under 1 second
- Memory usage under 200MB for typical documents

## 10. Platform Support

- Windows 10/11
- macOS 12+
- Linux (Ubuntu 22.04+, Fedora 38+)

## 11. Future Considerations (Not in v1)

- Batch processing (multiple files)
- Search within extracted text
- PDF preview alongside extracted text
- Page range selection for partial extraction
- Export to Markdown format
